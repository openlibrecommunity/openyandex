BinDiff-first Patch Map Workflow
- First objective is a complete change map, not implementation.
- Use BinDiff/Ghidra/rizin/source-path/string/resource evidence to identify everything Yandex changed.
- Split evidence into patch-map families.
- Triage each family:
  - keep: user/admin/network/product-visible behavior.
  - drop: pure third-party/runtime/media/vendor rebuild with no kept feature impact.
  - defer: useful but not first-wave.
  - unknown: needs more evidence.
- Assign subagents by logical family from .workflows/active-stage/subagent-backlog.txt.
- Implementation starts only after a family is mapped enough to name Chromium touch points.

User-visible Keep Scope
- UI, UX, branding, NTP, smartbox, settings, toolbar, menus, bubbles, widgets.
- Telemetry, metrics, tracking, experiments, ABT, logging.
- GOST, crypto, certificates, TLS/security behavior.
- AI: Alice, SpeechKit, YaGPT, neuro widgets, OCR/text actions.
- Privacy: AntiTracking, Adblock, NoReferer, custom headers.
- Enterprise: policies, DLP, device posture, browser enforcement, cloud management.
- Account: Passport, Yandex ID, sync/account UX.
- Update/onboarding/service behavior when product-visible.

Drop/Defer Scope
- Video/media engine changes unless they support a kept feature.
- FFmpeg/media stack rebuilds without product-visible impact.
- GPU/ANGLE/SwiftShader/vendor runtime rebuilds.
- Pure third-party rebuilt libraries.
- Crash/WER internals unless needed for telemetry/product behavior.
- Compatibility glue without kept-feature impact.
